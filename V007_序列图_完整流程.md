## 📊 带泳道的序列图（完整流程）

本文档展示了 V007 系统从用户请求到响应的完整交互流程，包括所有关键组件的交互细节。V007 在 V006 的基础上增加了真实数据访问和三温两湿图生成功能。

### 场景：用户请求生成粮情分析报告

```mermaid
sequenceDiagram

    participant FastAPI as ⚡ FastAPI<br/>main.py
    participant Middleware as 🔧 HTTP中间件
    participant AgentEP as 📡 Agent端点<br/>agent.py
    participant Schemas as 📋 Schemas<br/>schemas.py

    Note over FastAPI,Middleware: ========== 中间件处理 ==========

    Note over AgentSvc: ========== Agent核心编排 ==========

                    Domain-->>DataLoader: WarehouseInfo 对象
                    DataLoader-->>WMSCli: WarehouseInfo (domain模型)
                    WMSCli-->>Tools: WarehouseInfo (domain模型)
                    
                    Tools->>WMSCli: get_grain_temperature(<br/>"1", start_time, end_time)
                    WMSCli->>DataLoader: query("1", start_time, end_time)
                    DataLoader->>DataFile: 从索引中查询时间段数据

                    Domain-->>DataLoader: List[GrainTempData] 对象
                    DataLoader-->>WMSCli: List[GrainTempData] (domain模型)
                    WMSCli-->>Tools: List[GrainTempData] (domain模型)
                    
                    Tools->>Tools: 解析 temp_values<br/>转换为 List[Reading]
                    Tools->>Domain: 构造 Reading 对象<br/>(sensor_id, timestamp, value, type)
                    Domain-->>Tools: List[Reading] (domain模型)

                    Tools->>Tools: 生成两湿图<br/>generate_two_humidity_chart("1", start_time, end_time)
                    Tools->>Tools: 生成温度曲线图<br/>visualization("1", "line", start_time, end_time)
                    Tools->>Tools: 保存图表到<br/>artifacts/charts/
                    
                    Tools->>AnalysisSvc: analysis("1")<br/>传入 List[Reading]
                    AnalysisSvc->>AnalysisSvc: analyze_temperature(<br/>silo_id, readings: List[Reading])
                    AnalysisSvc->>AnalysisSvc: 计算统计值<br/>识别热点<br/>评估风险等级
                    AnalysisSvc->>Schemas: 构造 AnalysisResult<br/>(silo_id, findings, risk_level, score)
                    Schemas-->>AnalysisSvc: AnalysisResult 对象
                    AnalysisSvc-->>Tools: AnalysisResult (schemas模型)

                    LLMSvc->>LLM: analyze(query, context)
                    LLM-->>LLMSvc: 返回分析结果<br/>{response, reasoning}
                    LLMSvc-->>Tools: LLMResult

                    Tools->>Tools: 插入图表<br/>doc.add_picture(...)
                end
                
                Tools->>Tools: 保存文档<br/>doc.save(out_path)
                Tools-->>LLMSvc: 返回报告路径<br/>{file_path, status: "generated"}

        else LLM 返回最终答案

    Note over AgentSvc: ========== 构建响应 ==========
    
    AgentSvc->>AgentSvc: 提取结果<br/>answer, reasoning, tool_calls
    AgentSvc->>AgentSvc: 推断意图<br/>intent = tool_calls[0]["tool"]
    AgentSvc->>AgentSvc: 记录对话历史<br/>conversation_history.append()
    AgentSvc-->>AgentEP: 返回响应字典
    
    Note over AgentEP,Schemas: ========== 响应返回 ==========
    
    AgentEP->>AgentEP: 添加时间戳和 trace_id
    AgentEP->>Schemas: 构造 AgentChatResponse<br/>(query, intent, answer, reasoning, tool_calls, ...)

    AgentEP-->>Middleware: 返回响应对象
    
    Middleware->>Middleware: 计算处理时间<br/>process_time
    Middleware->>Middleware: 添加响应头<br/>X-Process-Time<br/>X-Request-ID
    Middleware->>Middleware: 记录日志<br/>"Request completed..."
    Middleware-->>FastAPI: 返回响应
    
    FastAPI-->>HTTP: HTTP 200 OK<br/>JSON 响应
    HTTP-->>User: 显示结果<br/>{answer, reasoning, tool_calls, ...}
```

## 📦 Model 层在流程中的位置和作用

### Domain 模型 (`domain.py`) - 业务数据核心结构

**位置：** 在数据获取和内部处理阶段

**作用：**
1. **WMSClient → Tools**: WMSClient 从 WMS API 获取 JSON 数据后，构造 Domain 模型
   - `WarehouseInfo`: 仓房基本信息（house_code, house_name, capacity 等）
   - `GrainTempData`: 粮温数据（temp_values, max_temp, min_temp 等）
   - `GasConcentrationData`: 气体浓度数据

2. **Tools → AnalysisService**: Tools 将 Domain 数据转换为内部处理格式
   - `Reading`: 传感器读数（sensor_id, timestamp, value, type）
   - 用于温度分析、热点识别等

**特点：**
- ✅ 描述业务概念（粮仓、传感器、温度数据）
- ✅ 用于内部业务逻辑处理
- ✅ 与 WMS 接口数据结构对应
- ✅ 相对稳定，不随 API 变化

**在序列图中的位置：**
```
WMS (JSON) → WMSClient → Domain (构造 WarehouseInfo/GrainTempData) → Tools
Tools → Domain (构造 Reading) → AnalysisService
```

---

### Schemas 模型 (`schemas.py`) - API 请求/响应格式

**位置：** 在 API 边界和 Service 层返回结果时

**作用：**
1. **API 请求验证** (`AgentChatRequest`):
   - FastAPI 自动验证 HTTP 请求体
   - 确保 `query`, `session_id`, `history` 字段格式正确
   - 位置：AgentEP 接收请求时

2. **API 响应序列化** (`AgentChatResponse`):
   - FastAPI 自动将 Python 对象序列化为 JSON
   - 包含 `query`, `intent`, `answer`, `reasoning`, `tool_calls` 等字段
   - 位置：AgentEP 返回响应时

3. **Service 层返回结构** (`AnalysisResult`):
   - AnalysisService 返回标准化的分析结果
   - 包含 `silo_id`, `findings`, `risk_level`, `score` 等字段
   - 位置：AnalysisService → Tools

**特点：**
- ✅ 描述 API 接口的输入输出
- ✅ 用于 HTTP 请求/响应的序列化和验证
- ✅ 可能组合多个 Domain 模型
- ✅ 可能包含 API 特有的字段（如 `trace_id`）

**在序列图中的位置：**
```
HTTP (JSON) → AgentEP → Schemas (AgentChatRequest 验证) → AgentService
AnalysisService → Schemas (AnalysisResult) → Tools
AgentService → AgentEP → Schemas (AgentChatResponse 序列化) → HTTP (JSON)
```

## 🔍 关键流程说明

### 1. 请求入口流程

```
用户 → HTTP客户端 → FastAPI → HTTP中间件 → Agent端点 → Schemas验证
```

- **FastAPI** (`main.py`): 应用入口，配置 CORS 和中间件
- **HTTP中间件**: 记录请求ID、处理时间，添加响应头
- **Agent端点** (`agent.py`): 唯一对外接口 `/api/v1/agent/chat`
- **Schemas** (`schemas.py`): FastAPI 自动验证请求（`AgentChatRequest`），序列化响应（`AgentChatResponse`）

### 2. Agent 核心编排流程

```
Agent端点 → AgentService → LLMService (Function Calling)
```

- **AgentService** (`agent_service.py`):
  - 构造 System Prompt
  - 构建消息列表
  - 调用 LLMService 进行 Function Calling
  - 提取结果并构建响应

### 3. LLM Function Calling 循环

```
LLMService → 通义千问 → 工具调用 → 工具执行 → 结果返回 → LLM生成答案
```

- **LLMService** (`llm_service.py`):
  - 支持最多3轮工具调用
  - 每轮：LLM 决定调用工具 → 执行工具 → 将结果返回给 LLM
  - 最终：LLM 生成包含工具结果的完整答案

### 4. 工具执行流程（以 T8 报告生成为例）

```
GrainTools.report() → WMSClient → WMS API
                   → AnalysisService
                   → LLMService (推理)
                   → 生成文档和图表
```

- **GrainTools** (`tools.py`): T1-T8 工具实现
- **WMSClient** (`wms_client.py`): 调用 WMS 标准数据接口
- **AnalysisService** (`analysis_service.py`): 粮情数据分析
- **LLMService**: 生成推理和建议

### 5. 数据流向

```
WMS数据接口 (JSON)
    ↓
WMSClient → Domain (构造 WarehouseInfo, GrainTempData)
    ↓
GrainTools → Domain (构造 Reading) → AnalysisService
    ↓
AnalysisService → Schemas (构造 AnalysisResult)
    ↓
LLMService → AgentService → Schemas (构造 AgentChatResponse)
    ↓
用户 (JSON)
```

**关键模型流转：**
- **Domain 模型** (`domain.py`): 业务数据核心结构，在 WMSClient → Tools → AnalysisService 之间传递
  - `WarehouseInfo`: 仓房基本信息
  - `GrainTempData`: 粮温数据
  - `Reading`: 传感器读数（内部处理）
- **Schemas 模型** (`schemas.py`): API 请求/响应格式，在 API 层和 Service 层之间传递
  - `AgentChatRequest`: 请求验证
  - `AgentChatResponse`: 响应序列化
  - `AnalysisResult`: 分析结果结构

## 🔄 多轮工具调用示例

以下场景展示了 LLM 如何进行多轮工具调用：

### 场景：用户查询 "1号仓的粮温情况如何？请给出储藏建议。"

**第1轮：**
- LLM 决定调用 `extraction` 获取数据
- 执行 `extraction("1", 24)`
- 返回粮温数据给 LLM

**第2轮：**
- LLM 基于数据调用 `analysis` 进行分析
- 执行 `analysis("1")`
- 返回分析结果给 LLM

**第3轮：**
- LLM 调用 `llm_reasoning` 生成建议
- 执行 `llm_reasoning(query, context)`
- 返回推理结果给 LLM

**最终：**
- LLM 综合所有工具结果，生成完整答案
- 返回给用户

## ⚠️ 错误处理流程

```mermaid
sequenceDiagram
    participant LLMSvc as LLMService
    participant LLM as 通义千问
    participant Tools as GrainTools
    
    LLMSvc->>LLM: chat.completions.create(...)
    
    alt LLM API 调用失败
        LLM-->>LLMSvc: 抛出异常
        LLMSvc->>LLMSvc: 捕获异常
        LLMSvc->>LLMSvc: 回退到 Mock LLM<br/>_analyze_mock()
        LLMSvc-->>AgentSvc: 返回 Mock 结果
    end
    
    LLMSvc->>Tools: 调用工具函数
    
    alt 工具执行失败
        Tools-->>LLMSvc: 抛出异常
        LLMSvc->>LLMSvc: 捕获异常
        LLMSvc->>LLMSvc: 构造错误响应<br/>{"error": str(e)}
        LLMSvc->>LLM: 将错误信息返回给 LLM
        LLM-->>LLMSvc: 生成包含错误信息的回答
    end
```

## 🎯 设计亮点

1. **Agent-Only 架构**: 单一入口，所有功能通过 LLM Function Calling 调用
2. **多轮工具调用**: 支持复杂的工具链调用（如：获取数据 → 分析 → 生成报告）
3. **错误容错**: LLM 失败时回退到 Mock，工具失败时返回错误信息给 LLM
4. **清晰分层**: API → Service → Tools → Integration
5. **完整工具集**: T1-T8 全部实现，支持 WMS 标准接口
6. **模型层分离**: Domain 模型（业务数据）和 Schemas 模型（API格式）职责清晰，Domain 在内部流转，Schemas 在 API 边界使用
