# V008 模型层详解：domain.py 与 schemas.py

## 📋 核心概念

在 V008 项目中，`domain.py` 和 `schemas.py` 遵循了**领域驱动设计（DDD）**和**API 设计分离**的原则，它们有不同的职责和使用场景。

---

## 🎯 两者的根本区别

### `domain.py` - 领域模型（Domain Models）

**职责：** 定义业务领域的核心实体和数据结构

**特点：**
- ✅ 描述**业务概念**（粮仓、传感器、温度数据等）
- ✅ 用于**内部业务逻辑**处理
- ✅ 与**数据库/WMS 接口**数据结构对应
- ✅ 相对稳定，不随 API 变化

### `schemas.py` - API Schema（请求/响应模型）

**职责：** 定义 API 接口的请求和响应格式

**特点：**
- ✅ 描述**API 接口**的输入输出
- ✅ 用于**HTTP 请求/响应**的序列化和验证
- ✅ 可能**组合多个领域模型**
- ✅ 可能包含**API 特有的字段**（如 trace_id）

---

## 📊 对比表格

| 维度 | `domain.py` | `schemas.py` |
|------|------------|-------------|
| **用途** | 业务逻辑处理 | API 接口定义 |
| **使用场景** | 内部服务、数据处理 | HTTP 请求/响应 |
| **稳定性** | 相对稳定 | 可能频繁变化 |
| **关注点** | "这是什么业务概念？" | "API 需要什么数据？" |
| **示例** | `Reading`, `WarehouseInfo` | `AgentChatRequest`, `AgentChatResponse` |

## 🔍 详细分析

### 1. `domain.py` - 领域模型详解

#### 1.1 原始基础模型（内部逻辑）

```python
# domain.py:6-28
class Reading(BaseModel):
    sensor_id: str
    timestamp: datetime
    value: float
    type: str  # 'temperature', 'humidity', 'gas'
    quality: str = "ok"

class Sensor(BaseModel):
    id: str
    type: str
    location: Dict[str, float]  # {'x': 1.0, 'y': 2.0, 'z': 3.0}

class Silo(BaseModel):
    id: str
    name: str
    capacity: float
    current_level: float
    sensors: List[Sensor] = []

class Warehouse(BaseModel):
    id: str
    name: str
    silos: List[Silo] = []
```

**用途：**
- 用于内部业务逻辑处理
- 描述业务实体之间的关系（仓库 → 仓 → 传感器 → 读数）
- 在 `tools.py`、`analysis_service.py` 中使用

**使用示例：**
```python
# tools.py:16
from app.models.domain import Reading

# 创建 Reading 对象用于分析
readings: List[Reading] = []
readings.append(Reading(
    sensor_id="1-T111",
    timestamp=datetime.now(),
    value=28.5,
    type="temperature"
))
```

#### 1.2 WMS 业务模型（对应接口规范）

```python
# domain.py:33-100
class WarehouseInfo(BaseModel):
    house_code: str
    house_name: str
    depot_name: str
    # ... 仓房详细信息

class GrainTempData(BaseModel):
    house_name: str
    check_time: str
    max_temp: float
    min_temp: float
    avg_temp: float
    temp_values: str  # "27.7,1,1,1|25.1,1,1,2"

class GasConcentrationData(BaseModel):
    house_name: str
    check_time: str
    avg_o2: float
    avg_ph3: float
    # ... 气体浓度数据
```

**用途：**
- 对应 WMS 标准数据接口的数据结构
- 用于从 WMS 获取数据时的数据模型
- 在 `wms_client.py` 中使用

**使用示例：**
```python
# wms_client.py:4
from app.models.domain import WarehouseInfo, GrainTempData

# WMS 客户端返回 WarehouseInfo 对象
def get_warehouse_info(self, house_code: str) -> WarehouseInfo:
    return WarehouseInfo(
        house_code=house_code,
        house_name=f"{house_code}号仓",
        depot_name="中央储备粮某直属库",
        # ...
    )
```

### 2. `schemas.py` - API Schema 详解

#### 2.1 API 请求模型（Request）

```python
# schemas.py:78-82
class AgentChatRequest(BaseModel):
    """Agent 对话请求"""
    query: str  # 用户的自然语言查询
    session_id: Optional[str] = None
    history: List[Dict[str, str]] = []
```

**用途：**
- 定义 API 接口的**输入格式**
- FastAPI 自动验证请求体
- 在 `agent.py` 端点中使用

**使用示例：**
```python
# agent.py:30-31
@router.post("/chat", response_model=AgentChatResponse)
def agent_chat(request: AgentChatRequest):
    # FastAPI 自动将 JSON 请求体转换为 AgentChatRequest 对象
    query = request.query  # 直接使用
```

#### 2.2 API 响应模型（Response）

```python
# schemas.py:84-92
class AgentChatResponse(BaseResponse):
    """Agent 对话响应"""
    query: str
    intent: str
    answer: str
    reasoning: str
    tool_calls: List[Dict[str, Any]]
    raw_results: Dict[str, Any]
    timestamp: datetime
    # trace_id 继承自 BaseResponse
```

**用途：**
- 定义 API 接口的**输出格式**
- FastAPI 自动序列化为 JSON
- 确保响应格式一致

**使用示例：**
```python
# agent.py:65
return AgentChatResponse(**response)
# FastAPI 自动将对象序列化为 JSON 响应
```

#### 2.3 Schema 引用 Domain 模型

```python
# schemas.py:4
from .domain import Reading, GrainTempData, GasConcentrationData, WarehouseInfo

# schemas.py:11-18
class GrainTempResponse(BaseResponse):
    data: List[GrainTempData]  # ← 引用 domain 模型

class WarehouseInfoResponse(BaseResponse):
    data: WarehouseInfo  # ← 引用 domain 模型
```

**设计模式：**
- Schema **组合** Domain 模型
- Schema 添加 API 特有的字段（如 `trace_id`）
- Domain 模型保持纯净，不包含 API 相关字段

## 🔄 两者的关系和协作

### 数据流转示例

```
1. HTTP 请求到达
   ↓
2. schemas.py: AgentChatRequest
   - 验证请求格式
   - 提取 query 字段
   ↓
3. 业务逻辑处理
   - 调用 WMS 接口
   - 返回 domain.py: WarehouseInfo, GrainTempData
   ↓
4. 数据处理和分析
   - 使用 domain.py: Reading 进行分析
   ↓
5. 构建响应
   - 组合 domain 模型数据
   - 添加 API 字段（trace_id, timestamp）
   ↓
6. schemas.py: AgentChatResponse
   - 序列化为 JSON
   - 返回给客户端
```

### 代码示例：完整流程

```python
# 1. API 端点接收请求（使用 schemas）
# agent.py:30
@router.post("/chat", response_model=AgentChatResponse)
def agent_chat(request: AgentChatRequest):  # ← schemas.py
    query = request.query
    
    # 2. 调用业务逻辑（使用 domain）
    # tools.py:133
    warehouse_info = self.wms_client.get_warehouse_info(silo_id)
    # 返回 domain.py: WarehouseInfo
    
    # 3. 数据处理（使用 domain）
    # tools.py:144
    readings_models: List[Reading] = []  # ← domain.py
    readings_models.append(Reading(...))
    
    # 4. 构建响应（使用 schemas）
    response = {
        "query": query,
        "answer": "...",
        # ...
    }
    return AgentChatResponse(**response)  # ← schemas.py
```

## 🎯 为什么需要分离？

### 1. 职责分离（Separation of Concerns）

```
domain.py  ← 关注"业务是什么"
schemas.py ← 关注"API 需要什么"
```

**好处：**
- 业务逻辑独立于 API 设计
- API 可以变化，不影响业务逻辑
- 代码更清晰，易于维护

### 2. 灵活性

**场景：** 如果 API 需要添加字段，只需要修改 `schemas.py`：

```python
# schemas.py:84
class AgentChatResponse(BaseResponse):
    query: str
    answer: str
    new_field: str  # ← 新增字段，不影响 domain.py
    # ...
```

**domain.py 保持不变：**
```python
# domain.py 不需要修改
# 业务逻辑不受影响
```

### 3. 复用性

**Domain 模型可以在多个地方使用：**

```python
# 1. 在 WMS 客户端中使用
wms_client.get_warehouse_info() → WarehouseInfo

# 2. 在工具中使用
tools.get_warehouse_info() → WarehouseInfo

# 3. 在 Schema 中组合
WarehouseInfoResponse.data: WarehouseInfo
```

## 📝 总结

### `domain.py` - 业务领域模型

**核心思想：** "这是什么业务概念？"

- ✅ 描述业务实体（仓房、传感器、温度数据）
- ✅ 用于内部业务逻辑处理
- ✅ 对应 WMS 接口数据结构
- ✅ 相对稳定，不随 API 变化

**使用位置：**
- `wms_client.py` - WMS 数据接口
- `tools.py` - 工具实现
- `analysis_service.py` - 分析服务

### `schemas.py` - API 接口模型

**核心思想：** "API 需要什么数据？"

- ✅ 定义 API 请求/响应格式
- ✅ 用于 HTTP 接口的序列化和验证
- ✅ 可能组合多个 domain 模型
- ✅ 可能包含 API 特有字段（trace_id）

**使用位置：**
- `agent.py` - API 端点
- `llm_service.py` - LLM 服务

### 两者的关系

```
schemas.py  ← 引用 → domain.py
    ↓              ↓
API 层        业务层
    ↓              ↓
HTTP 接口    业务逻辑
```

**设计原则：**
- Domain 模型：**稳定**，描述业务本质
- Schema 模型：**灵活**，适应 API 需求
- Schema 可以**组合** Domain 模型
- Schema 可以**添加** API 特有字段

## 🎓 类比理解

### 类比 1：餐厅点餐（简单理解）

**`domain.py` = 菜单（菜品定义）**
- 描述菜品本身（宫保鸡丁、麻婆豆腐）
- 不关心谁来点、怎么点
- 相对稳定

**`schemas.py` = 订单单（点餐格式）**
- 定义点餐的格式（菜品名称、数量、备注）
- 定义订单的格式（订单号、总价、时间）
- 可能变化（新增优惠券字段）

**关系：**
- 订单单引用菜单中的菜品
- 订单单添加订单特有的信息（订单号、时间）

---

### 类比 2：数据库三级模式（您的第一个类比）

**`domain.py` ≈ 模式（Conceptual Schema）**
- 描述数据库的整体逻辑结构
- 定义业务实体的核心属性
- 相对稳定，不随用户视图变化

**`schemas.py` ≈ 外模式（External Schema）**
- 不同用户/应用看到不同的数据视图
- API 是"用户"，看到的是 API 视图
- 可能包含模式中没有的字段（如 trace_id）

**适用性评价：** ✅ **非常合适！**

**优点：**
- ✅ 准确描述了"视图"的概念：schemas 是 API 的视图
- ✅ 体现了稳定性差异：模式稳定，外模式可能变化
- ✅ 体现了组合关系：外模式基于模式，但可能添加字段

**局限性：**
- ⚠️ 数据库外模式通常是模式的子集或投影
- ⚠️ 而 schemas 可能包含 domain 没有的字段（如 trace_id）
- ⚠️ 但整体概念非常接近！

**修正理解：**
```
domain.py (模式) → 业务数据的核心结构
    ↓
schemas.py (外模式) → API 的视图
    - 可能包含 domain 的字段（组合）
    - 可能添加 API 特有字段（trace_id）
    - 可能只选择部分字段（投影）
```

---

### 类比 3：网络协议分层（您的第二个类比）

**`domain.py` ≈ 物理层数据**
- 原始的业务数据
- 不包含协议相关的信息

**`schemas.py` ≈ 数据链路层帧**
- 添加了"帧头"（trace_id、timestamp）
- 封装了 domain 数据

**适用性评价：** ⚠️ **部分合适，但有局限性**

**优点：**
- ✅ 体现了"封装"的概念：schemas 封装 domain
- ✅ 体现了"添加元数据"：schemas 添加 API 字段

**局限性：**
- ❌ 物理层和数据链路层是**上下层关系**（数据链路层封装物理层）
- ❌ 但 schemas 和 domain 更像是**组合关系**（schemas 组合 domain，而不是封装）
- ❌ 物理层是底层，数据链路层在上层
- ❌ 但 domain 不是"底层"，schemas 也不是"上层"，它们是不同关注点

**更准确的网络类比：**

**`domain.py` ≈ 应用层数据（业务数据）**
- 业务逻辑的数据结构
- 不关心传输格式

**`schemas.py` ≈ 表示层（Presentation Layer）**
- 定义数据的表示格式（JSON Schema）
- 添加传输相关的元数据（trace_id）
- 负责序列化/反序列化

**或者：**

**`domain.py` ≈ 业务数据（Payload）**
- 实际要传输的业务数据

**`schemas.py` ≈ 协议头 + 业务数据（Protocol Header + Payload）**
- 协议头：trace_id, timestamp（API 元数据）
- 业务数据：domain 模型的数据

```
┌─────────────────────────┐
│  schemas.py              │
│  ┌───────────────────┐  │
│  │ API 元数据        │  │ ← trace_id, timestamp
│  │ (协议头)          │  │
│  └───────────────────┘  │
│  ┌───────────────────┐  │
│  │ domain 数据       │  │ ← WarehouseInfo, etc.
│  │ (业务数据)        │  │
│  └───────────────────┘  │
└─────────────────────────┘
```
