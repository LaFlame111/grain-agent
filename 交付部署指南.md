# 粮情分析智能体 (Grain Agent V010) — 交付部署指南

## 一、交付物清单

| 文件/目录 | 内容 | 大小 |
|-----------|------|------|
| 项目根目录 | Agent 后端源码 + 前端 + 知识库原始资料 + 测试集 | ~200 MB |
| `ragflow_docker/` | RAGFlow 部署配置（docker-compose.yml 等） | ~5 MB |
| `ragflow_export/` | Docker 镜像 + 已解析知识库数据卷（免拉取、免解析） | ~4.4 GB |

### ragflow_export 包内容

| 文件 | 大小 | 说明 |
|------|------|------|
| `images.tar` (或 `.part-aa/ab/ac` 分卷) | ~4.0 GB | 5 个 Docker 镜像（RAGFlow v0.24.0 + MySQL + ES + MinIO + Redis） |
| `vol_mysql.tar.gz` | ~16 MB | MySQL 数据（用户账号、数据集配置、文档元信息） |
| `vol_es.tar.gz` | ~76 MB | Elasticsearch 数据（向量索引，4001 个 chunks） |
| `vol_minio.tar.gz` | ~339 MB | MinIO 数据（上传的原始文件副本） |
| `vol_redis.tar.gz` | ~16 KB | Redis 缓存数据 |

> 如果通过 `git clone` 获取项目，`ragflow_export/` 不在仓库中，部署脚本会自动从 GitHub Releases 下载。
> 如果通过整包拷贝（U 盘/网盘）获取项目，`ragflow_export/` 已就位，脚本会跳过下载步骤。

## 三、一键部署（推荐）

### 方式 A：Windows（setup.ps1）

```
1. git clone https://gitcode.com/yekindarly/main.git grain-agent
2. 右键 setup.ps1 → "以管理员身份运行"
3.（若首次启用 WSL2）重启后再运行一次 setup.ps1
4. 等待自动完成（5~15 分钟）
5. 按提示输入 DASHSCOPE_API_KEY（通义千问 API Key）
6. 脚本自动获取 RAGFlow API Key 和数据集 ID 并写入 .env
7. 运行 .\start_server.ps1 启动 Agent
8. 浏览器打开 http://127.0.0.1:8000/ui
```

`setup.ps1` 自动完成（11 步）：

| 步骤 | 内容 | 已完成时 |
|------|------|----------|
| 0 | 管理员权限提升 | 已是管理员 → 跳过 |
| 1 | 检测/启用 WSL2 | 已启用 → 跳过 |
| 2 | 检测/安装 Docker Desktop | 已运行 → 跳过 |
| 3 | 配置 WSL2 内存（12GB） | 已配置 → 跳过 |
| 4 | 检测/安装 Python 3.10+ | 已安装 → 跳过 |
| 5 | 从 GitHub Releases 下载镜像和数据 | 文件已在本地 → 跳过 |
| 6 | 导入 Docker 镜像 | 镜像已存在 → 跳过 |
| 7 | 首次启动 Docker Compose（创建数据卷） | 卷已存在 → 跳过 |
| 8 | 导入知识库数据到数据卷 | 已有数据 → 跳过 |
| 9 | 启动 RAGFlow 服务 | docker compose up（幂等） |
| 10 | 安装 Python 依赖 + 配置 .env | pip 跳过已装包；已有 Key → 跳过 |
| 11 | 输出总结 | 始终显示 |

### 方式 B：Linux（setup.sh）

```bash
git clone https://gitcode.com/yekindarly/main.git grain-agent
cd grain-agent
chmod +x setup.sh
sudo ./setup.sh
# 按提示输入 DASHSCOPE_API_KEY
# 完成后：
bash start.sh
# 浏览器打开 http://<服务器IP>:8000/ui
```

`setup.sh` 自动完成（9 步）：

| 步骤 | 内容 | 已完成时 |
|------|------|----------|
| 1 | 检测/安装 Docker Engine | 已运行 → 跳过 |
| 2 | 检测/安装 Python 3.10+ | 已安装 → 跳过 |
| 3 | 从 GitHub Releases 下载镜像和数据 | 文件已在本地 → 跳过 |
| 4 | 导入 Docker 镜像 | 镜像已存在 → 跳过 |
| 5 | 首次启动 Docker Compose（创建数据卷） | 卷已存在 → 跳过 |
| 6 | 导入知识库数据到数据卷 | 已有数据 → 跳过 |
| 7 | 启动 RAGFlow 服务 | docker compose up（幂等） |
| 8 | 安装 Python 依赖 + 配置 .env | 同上 |
| 9 | 输出总结 | 始终显示 |

### 公共特性

- **幂等**：每一步先检测是否已完成，重复运行安全，不会破坏已有环境
- **断点续传**：下载中断后重新运行，已下载的文件自动跳过
- **日志文件**：部署过程完整记录在 `setup.log`，出问题时可发此文件排查
- **Windows 重启恢复**：WSL2 启用后需要重启，`.setup_state` 文件记录进度，重启后自动续跑

## 四、手动部署（备用）

如果一键脚本遇到问题，可按以下步骤手动部署。前提：已安装 Docker 和 Python 3.10+。

### 4.1 导入 Docker 镜像

```bash
# 如果只有分卷文件，先合并
cat ragflow_export/images.tar.part-* > ragflow_export/images.tar

# 导入镜像（约 2-5 分钟）
docker load -i ragflow_export/images.tar

# 验证（应看到 5 个镜像）
docker images | grep -E "ragflow|mysql|elasticsearch|minio|valkey"
```

### 4.2 启动 RAGFlow 创建数据卷

```bash
cd ragflow_docker
docker compose up -d
# 等待 MySQL 容器 healthy（约 1-2 分钟）
docker compose ps
# 停止容器（保留数据卷）
docker compose down
cd ..
```

### 4.3 导入知识库数据

使用 RAGFlow 镜像（不依赖 alpine）：

```bash
# Linux / Git Bash
IMG="infiniflow/ragflow:v0.24.0"
EXPORT="$(cd ragflow_export && pwd)"

docker run --rm -v "ragflow_docker_mysql_data:/data" -v "$EXPORT:/backup:ro" $IMG sh -c "cd /data && tar xzf /backup/vol_mysql.tar.gz"
docker run --rm -v "ragflow_docker_esdata01:/data"   -v "$EXPORT:/backup:ro" $IMG sh -c "cd /data && tar xzf /backup/vol_es.tar.gz"
docker run --rm -v "ragflow_docker_minio_data:/data"  -v "$EXPORT:/backup:ro" $IMG sh -c "cd /data && tar xzf /backup/vol_minio.tar.gz"
docker run --rm -v "ragflow_docker_redis_data:/data"  -v "$EXPORT:/backup:ro" $IMG sh -c "cd /data && tar xzf /backup/vol_redis.tar.gz"
```

```powershell
# Windows PowerShell
$img = "infiniflow/ragflow:v0.24.0"
$export = (Resolve-Path ".\ragflow_export").Path

docker run --rm -v "ragflow_docker_mysql_data:/data" -v "${export}:/backup:ro" $img sh -c "cd /data && tar xzf /backup/vol_mysql.tar.gz"
docker run --rm -v "ragflow_docker_esdata01:/data"   -v "${export}:/backup:ro" $img sh -c "cd /data && tar xzf /backup/vol_es.tar.gz"
docker run --rm -v "ragflow_docker_minio_data:/data"  -v "${export}:/backup:ro" $img sh -c "cd /data && tar xzf /backup/vol_minio.tar.gz"
docker run --rm -v "ragflow_docker_redis_data:/data"  -v "${export}:/backup:ro" $img sh -c "cd /data && tar xzf /backup/vol_redis.tar.gz"
```

> 数据卷名称前缀 `ragflow_docker_` 必须与 Docker Compose 项目名一致。

### 4.4 启动 RAGFlow

```bash
cd ragflow_docker
docker compose up -d
# 等待所有容器 healthy（约 1-2 分钟）
docker compose ps
```

验证：访问 http://localhost:80 可看到 RAGFlow Web UI。

### 4.5 配置 Agent

```bash
# 安装 Python 依赖
pip install -r requirements.txt

# 创建 .env 配置
cp .env.example .env

# 获取 RAGFlow API Key 和数据集 ID（自动）
python setup_apitoken.py

# 编辑 .env，填入：
# DASHSCOPE_API_KEY=你的通义千问API_Key（手动申请: https://dashscope.console.aliyun.com/）
# RAGFLOW_API_KEY=ragflow-xxxx（上一步输出）
# RAGFLOW_DATASET_IDS=xxx（上一步输出）
```

### 4.6 启动 Agent

```powershell
# Windows PowerShell
.\start_server.ps1

# Linux / Git Bash
bash start.sh
```

## 七、系统架构

```
用户 → FastAPI (port 8000)
         ├── 前端 /ui（聊天界面）
         ├── AgentService（意图识别 + 工具调用）
         │     ├── GrainTools (T1-T8)：查询/分析/预测/图表/报告
         │     │     └── WMS Client → WMS API
         │     ├── knowledge_search → RAGFlow (port 9380)
         │     └── LLM Service → 通义千问 API
         └── 静态文件 /artifacts/（生成的图表和报告）
```

### 核心模块

| 文件 | 职责 |
|------|------|
| `app/main.py` | FastAPI 入口 |
| `app/services/agent_service.py` | Agent 主服务：意图识别 + 工具调度 |
| `app/services/tools.py` | 粮情工具集 T1-T8（查询/分析/预测/图表/报告） |
| `app/services/tool_definitions.py` | 工具 JSON Schema 定义（供 LLM Function Calling） |
| `app/services/rag_service.py` | RAGFlow 知识检索服务（三层架构 + rerank） |
| `app/services/llm_service.py` | LLM 服务（通义千问，含 Function Calling） |
| `app/services/wms_client.py` | WMS 接口客户端 |
| `app/core/config.py` | 配置管理（从 .env 加载） |

### 关键配置

| 文件 | 说明 |
|------|------|
| `.env` | 环境变量（API Key、RAGFlow 配置、RAG 参数） |
| `.env.example` | 配置模板（含详细注释） |
| `ragflow_docker/` | RAGFlow Docker Compose 配置 |
| `data/knowledge/` | 知识库原始资料（167 个粮储标准文档，备份用） |

## 八、常见问题

### 部署问题

#### Q1: setup.ps1 运行后提示需要重启

首次启用 WSL2 需要重启一次。重启后再次右键运行 `setup.ps1`，脚本会从 `.setup_state` 文件自动识别并从断点继续。

#### Q2: Docker Desktop 启动超时（Windows）

- 确认 BIOS 中虚拟化（VT-x/AMD-V）已启用
- 手动打开 Docker Desktop，等待其完全启动后重新运行 `setup.ps1`
- 检查 Windows 功能中"适用于 Linux 的 Windows 子系统"和"虚拟机平台"是否已启用

#### Q3: Docker Engine 安装失败（Linux）

- `setup.sh` 支持 apt（Ubuntu/Debian）和 yum（CentOS/RHEL），其他发行版需手动安装
- 手动安装参考: https://docs.docker.com/engine/install/
- 安装后运行 `sudo systemctl start docker` 启动

#### Q4: GitHub Releases 下载太慢或失败

- 脚本使用 `curl` 断点续传（`-C -`）和 3 次重试
- 如果网络实在不行，可手动下载 Release 附件到 `ragflow_export/` 目录后重新运行脚本
- GitCode Releases 地址: https://gitcode.com/yekindarly/main/releases/tag/v2.0.0

### 运行问题

#### Q5: RAGFlow 容器启动失败

- **Windows**: 确认 WSL2 内存配置至少 12GB（检查 `%USERPROFILE%\.wslconfig`）
- **Linux**: 确认系统内存 ≥ 16GB（`free -h`）
- 查看日志：`cd ragflow_docker && docker compose logs ragflow`
- 确认镜像已导入：`docker images | grep ragflow`

#### Q6: 知识库为空（数据导入后无数据）

- 确认数据卷名称前缀正确（`ragflow_docker_`，而非 `docker_`）
- 数据导入通过 `docker run` 执行，与容器运行状态无关
- 重新运行 `setup.ps1` 或 `setup.sh` 会自动检测并处理

#### Q7: RAGFlow 可用但知识检索无结果

- 确认 `.env` 中 `RAGFLOW_DATASET_IDS` 与 Web UI 中数据集 ID 一致
- 确认 `RAGFLOW_API_KEY` 正确（格式：`ragflow-xxxx`）
- 在 Web UI (http://localhost:80) 进入数据集页面确认文档解析状态正常

#### Q8: Agent 回答不引用知识库

- 确认 RAGFlow 5 个容器全部运行：`cd ragflow_docker && docker compose ps`
- 测试 RAGFlow API 是否可达：`curl http://127.0.0.1:9380/api/v1/datasets`
- 查看 `server.log` 中是否有 RAGFlow 调用报错

#### Q9: Docker 磁盘空间不足

- RAGFlow 完整部署约占 10-15GB
- **Windows**: 默认 Docker 数据在 C 盘，可在 Docker Desktop Settings → Resources → Disk image location 迁移
- **Linux**: Docker 数据默认在 `/var/lib/docker/`，可通过修改 `/etc/docker/daemon.json` 的 `data-root` 迁移

#### Q10: WMS 接口超时

- 确认 WMS 服务器可达
- 超时时间可在 `app/core/wms_endpoints.py` 中调整（默认 5 秒）
- 如网络较慢可适当调大

#### Q11: PDF 文档解析乱码

- 部分 PDF（如扫描件）需要 OCR 解析模式
- 在 RAGFlow Web UI 中对该文档切换解析方法为 OCR 模式重新解析
