## 3. 关键发现

### 3.1 enhanced 为什么比 basic 差？

enhanced 模式同时改了 4 个变量（query 注入 + 宽阈值 + 大 page_size + reranker），其中 **关键词注入是主要问题**：

- 把"低温储藏的温度要求"变成"低温储藏的温度要求 粮温 仓温 粮堆温度 GB/T 29890"
- 注入的关键词污染了 BM25 侧的匹配，大量不相关 chunk 被召回
- Context Precision 暴跌 -24.3%，连带拖垮了 Faithfulness 和 Answer Relevancy

**结论**: RAGFlow 本身已有 Hybrid Search，在 query 上做加法是画蛇添足。

### 3.2 rerank_clean 为什么全面最优？

rerank_clean 的策略是 **干净 query + 广召回 + reranker 精排**：

1. **不修改 query** — 避免引入噪声
2. **宽阈值 (0.1) + 大 page_size (15)** — 尽可能多召回候选（Context Recall 达到满分 1.0）
3. **gte-rerank 精排** — 从 15 个候选中选出最优 5 个（Context Precision 提升至 0.7928）
4. **LLM 拿到更好的 context** — Faithfulness 0.9939 接近完美，Answer Relevancy +6.1%

### 3.3 仍存在的问题

部分问题在所有模式下都回答失败（basic 中也是 0 分），这不是检索策略的问题，而是 **分块质量问题**：

| 失败问题 | 缺失信息 | 原因 |
|----------|----------|------|
| 低温储藏温度要求 (15℃/20℃/25℃) | 8.1 节具体数值 | chunk 边界切断 |
| 粮温≤15℃ 检测周期 | 7.1.1.2.1 条款 | 相邻条款分属不同 chunk |
| CO2 气调浓度 35%/15天 | 8.3.1.3 段落 | 被切到其他 chunk |
| 严重虫粮处理时限 7天/3天 | 9.1.2 条款 | 召回不完整 |

当前 `chunk_token_num=512`，对于标准规范文档偏小（条款之间紧密关联，需要一起切）。

## 4. 待测试实验

### 4.1 阈值敏感性实验（rerank 系列）

已在脚本中注册，尚未运行：

| 模式 | threshold | page_size | reranker | 目的 |
|------|-----------|-----------|----------|------|
| rerank_clean | 0.1 | 15 | gte-rerank | 已完成，基准 |
| **rerank_t02** | **0.2** | 15 | gte-rerank | 中等阈值，看精准率能否提升 |
| **rerank_t03** | **0.3** | 15 | gte-rerank | 与 basic 同阈值，纯 reranker 增益 |

运行命令：
```bash
cd V008
python -m scripts.eval_ablation --mode rerank_t02
python -m scripts.eval_ablation --mode rerank_t03
```

如果 RAGAS 评估出现 nan（LLM Judge 限频），用 `--eval-only` 重跑评估：
```bash
python -m scripts.eval_ablation --mode rerank_t02 --eval-only
```

**预期**：
- threshold 提高 → Context Precision 可能进一步提升
- 但 Context Recall 可能从 1.0 下降（候选被阈值过滤掉）
- 最佳 threshold 应在精准率和召回率之间找到平衡点

### 4.2 更强 reranker 模型（后续可选）

RAGFlow 内置只有 gte-rerank。如果需要更强的 reranker，两种方案：

**方案 A: 在代码中直接调 DashScope rerank API（绕过 RAGFlow）**
```python
# 流程: RAGFlow 广召回(不带reranker) → 外部 rerank API 精排
resp = httpx.post(
    "https://dashscope.aliyuncs.com/api/v1/services/rerank/text-rerank/text-rerank",
    headers={"Authorization": f"Bearer {DASHSCOPE_API_KEY}"},
    json={"model": "gte-rerank", "input": {"query": q, "documents": docs}},
)
```

**方案 B: 部署更强的 reranker 模型到 RAGFlow**
- BAAI/bge-reranker-v2-m3（多语言，中文效果好）
- netease-youdao/bce-reranker-base_v1（中文专精）
- 需要在 RAGFlow 模型管理中添加

### 4.3 分块优化（治本）

将 chunk_token_num 从 512 调到 1024，重新解析文档，解决条款被切断的问题。

## 5. 当前最佳配置

基于实验数据，生产环境推荐配置：

```
query:                    原始用户 query，不做任何修改
similarity_threshold:     0.1（或待测试后的最佳值）
vector_similarity_weight: 0.3
page_size:                15
rerank_id:                gte-rerank
top_k:                    5
```

对应 `.env` 修改：
```
RAG_ENABLE_QUERY_REWRITE=false
RAG_SIMILARITY_THRESHOLD=0.1
RAG_TOP_K=5
```

生产代码 `rag_service.py` 待更新为 rerank_clean 策略（等阈值实验完成后一并更新）。

## 6. 脚本模块说明

| 模块 | 功能 |
|------|------|
| `retrieve_no_rag()` | 返回空列表，纯 LLM |
| `retrieve_basic()` | 直接调 RAGFlow API，默认参数 |
| `retrieve_enhanced()` | 关键词注入 + rerank + 参数调优（已证实效果差） |
| `retrieve_rerank_clean()` | 干净 query + 广召回 + rerank (threshold=0.1) |
| `retrieve_rerank_t02()` | 干净 query + 广召回 + rerank (threshold=0.2) |
| `retrieve_rerank_t03()` | 干净 query + 广召回 + rerank (threshold=0.3) |
| `generate_answer()` | LLM 回答生成 |
| `run_ragas_evaluation()` | RAGAS 评估 |
| `print_comparison_table()` | 多组对比表格 + 提升百分比 |

使用方式：
```bash
cd V008
python -m scripts.eval_ablation --mode all          # 全部模式
python -m scripts.eval_ablation --mode rerank_t02   # 单跑一组
python -m scripts.eval_ablation --mode rerank_t02 --eval-only  # 仅重跑评估
```

增量保存：每组跑完立即写入 `data/eval/ablation_results.json`，中断不丢失。
