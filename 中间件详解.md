# V006 中间件详解

## 📋 概述

V006 系统在 `app/main.py` 中配置了两个中间件，用于处理 HTTP 请求和响应。本文档详细解释这两个中间件的作用、工作原理和使用场景。

---

## 🔧 中间件 1: CORS 中间件

### 代码位置
```python
# V006/app/main.py 第27-33行
app.add_middleware(
    CORSMiddleware,
    allow_origins=["*"],  # 允许所有来源（开发环境）
    allow_credentials=True,
    allow_methods=["*"],
    allow_headers=["*"],
)
```

### 什么是 CORS？

**CORS (Cross-Origin Resource Sharing)** 是浏览器的安全机制，用于控制跨域请求。

**跨域问题场景：**
- 前端运行在 `http://localhost:3000`
- 后端 API 运行在 `http://localhost:8000`
- 浏览器会阻止这种跨域请求（协议、域名、端口任一不同即为跨域）

### CORS 中间件的作用

**1. 允许跨域请求**
- 解决前端和后端不在同一域名/端口的访问问题
- 允许浏览器发送跨域请求到您的 API

**2. 配置项说明**

| 配置项 | 值 | 说明 |
|--------|-----|------|
| `allow_origins` | `["*"]` | 允许所有来源的请求<br/>⚠️ **生产环境应限制为具体域名**，如 `["http://localhost:3000", "https://yourdomain.com"]` |
| `allow_credentials` | `True` | 允许携带认证信息（如 Cookies） |
| `allow_methods` | `["*"]` | 允许所有 HTTP 方法（GET, POST, PUT, DELETE 等） |
| `allow_headers` | `["*"]` | 允许所有请求头 |

### 工作原理

**1. 预检请求（OPTIONS）**
```
浏览器 → API: OPTIONS /api/v1/agent/chat
         Headers: Origin: http://localhost:3000
                  Access-Control-Request-Method: POST
         
API → 浏览器: 200 OK
         Headers: Access-Control-Allow-Origin: *
                  Access-Control-Allow-Methods: *
                  Access-Control-Allow-Headers: *
```

**2. 实际请求**
```
浏览器 → API: POST /api/v1/agent/chat
         Headers: Origin: http://localhost:3000
         
API → 浏览器: 200 OK
         Headers: Access-Control-Allow-Origin: *
                  (响应数据)
```

### 使用场景

✅ **适用场景：**
- 前端和后端分离部署
- 开发环境测试
- 需要从不同域名访问 API

❌ **不适用场景：**
- 纯后端服务（不涉及浏览器）
- 同源请求（前端和后端在同一域名）

### 安全建议

**开发环境（当前配置）：**
```python
allow_origins=["*"]  # 允许所有来源，方便开发
```

**生产环境（推荐配置）：**
```python
allow_origins=[
    "https://yourdomain.com",
    "https://www.yourdomain.com"
]  # 只允许特定域名
```

## ⏱️ 中间件 2: 自定义 HTTP 中间件（请求追踪和性能监控）

### 代码位置
```python
# V006/app/main.py 第35-54行
@app.middleware("http")
async def add_process_time_header(request: Request, call_next):
    request_id = str(uuid.uuid4())
    start_time = time.time()
    
    logger.info(f"Request started: {request.method} {request.url} [ID: {request_id}]")
    
    try:
        response = await call_next(request)
        
        process_time = time.time() - start_time
        response.headers["X-Process-Time"] = str(process_time)
        response.headers["X-Request-ID"] = request_id
        
        logger.info(f"Request completed: {request.method} {request.url} [ID: {request_id}] - Took {process_time:.4f}s")
        
        return response
    except Exception as e:
        logger.error(f"Request failed: {e}", exc_info=True)
        raise
```

### 中间件的作用

这个自定义中间件提供了**请求追踪**和**性能监控**功能：

#### 1. **生成唯一请求ID**
```python
request_id = str(uuid.uuid4())
```
- 为每个请求生成唯一的 UUID
- 用于追踪和关联日志

**用途：**
- 在分布式系统中追踪请求
- 关联请求的日志和错误信息
- 调试时快速定位问题

#### 2. **记录请求开始时间**
```python
start_time = time.time()
```
- 记录请求进入中间件的时间戳
- 用于计算请求处理耗时

#### 3. **记录请求开始日志**
```python
logger.info(f"Request started: {request.method} {request.url} [ID: {request_id}]")
```
**日志示例：**
```
2025-12-18 19:00:00 - INFO - Request started: POST /api/v1/agent/chat [ID: 9fff2c98-509a-499f-95bc-9fd7a9b0442e]
```

#### 4. **执行实际请求处理**
```python
response = await call_next(request)
```
- `call_next(request)` 调用下一个中间件或路由处理器
- 这是中间件的核心：**在请求前后执行代码**

#### 5. **计算处理时间**
```python
process_time = time.time() - start_time
```
- 计算从请求开始到响应返回的总耗时
- 单位：秒（浮点数，精确到微秒）

#### 6. **添加响应头**
```python
response.headers["X-Process-Time"] = str(process_time)
response.headers["X-Request-ID"] = request_id
```

**响应头说明：**

| 响应头 | 示例值 | 说明 |
|--------|--------|------|
| `X-Process-Time` | `2.3456` | 请求处理耗时（秒）<br/>客户端可以读取此值监控 API 性能 |
| `X-Request-ID` | `9fff2c98-509a-499f-95bc-9fd7a9b0442e` | 请求唯一标识<br/>客户端可以记录此 ID，用于问题追踪 |

#### 7. **记录请求完成日志**
```python
logger.info(f"Request completed: {request.method} {request.url} [ID: {request_id}] - Took {process_time:.4f}s")
```
**日志示例：**
```
2025-12-18 19:00:02 - INFO - Request completed: POST /api/v1/agent/chat [ID: 9fff2c98-509a-499f-95bc-9fd7a9b0442e] - Took 2.3456s
```

#### 8. **错误处理**
```python
except Exception as e:
    logger.error(f"Request failed: {e}", exc_info=True)
    raise
```
- 捕获请求处理过程中的异常
- 记录错误日志（包含完整堆栈信息）
- 重新抛出异常，让 FastAPI 的错误处理器处理

### 工作流程图示

```
请求进入
    ↓
生成 request_id (UUID)
    ↓
记录 start_time
    ↓
记录日志: "Request started..."
    ↓
调用 call_next(request) → 执行路由处理器
    ↓
计算 process_time = time.time() - start_time
    ↓
添加响应头: X-Process-Time, X-Request-ID
    ↓
记录日志: "Request completed... Took X.XXs"
    ↓
返回响应
```

### 实际使用示例

**1. 客户端发送请求：**
```bash
curl -X POST http://localhost:8000/api/v1/agent/chat \
  -H "Content-Type: application/json" \
  -d '{"query": "请生成1号仓的日报报告"}'
```

**2. 服务器日志输出：**
```
2025-12-18 19:00:00 - INFO - Request started: POST /api/v1/agent/chat [ID: 9fff2c98-509a-499f-95bc-9fd7a9b0442e]
2025-12-18 19:00:02 - INFO - Request completed: POST /api/v1/agent/chat [ID: 9fff2c98-509a-499f-95bc-9fd7a9b0442e] - Took 2.3456s
```

**3. 响应头示例：**
```http
HTTP/1.1 200 OK
Content-Type: application/json
X-Process-Time: 2.3456
X-Request-ID: 9fff2c98-509a-499f-95bc-9fd7a9b0442e

{
  "query": "请生成1号仓的日报报告",
  "answer": "...",
  ...
}
```

**4. 客户端读取响应头：**
```python
import requests

response = requests.post(
    "http://localhost:8000/api/v1/agent/chat",
    json={"query": "请生成1号仓的日报报告"}
)

# 读取处理时间
process_time = response.headers.get("X-Process-Time")
print(f"API 处理耗时: {process_time} 秒")

# 读取请求ID（用于问题追踪）
request_id = response.headers.get("X-Request-ID")
print(f"请求ID: {request_id}")
```

### 使用场景

✅ **适用场景：**
- **性能监控**：监控 API 响应时间
- **问题追踪**：通过 request_id 关联日志和错误
- **调试**：快速定位慢请求
- **运维**：分析 API 性能趋势

✅ **实际应用：**
- 前端可以读取 `X-Process-Time` 显示"处理中..."提示
- 运维人员可以通过 `X-Request-ID` 在日志中追踪特定请求
- 监控系统可以收集 `X-Process-Time` 进行性能分析

### 中间件执行顺序

FastAPI 中间件的执行顺序是**后进先出（LIFO）**：

```
请求进入
    ↓
CORS 中间件（第27行添加）
    ↓
自定义 HTTP 中间件（第35行添加）
    ↓
路由处理器（agent_chat）
    ↓
自定义 HTTP 中间件（返回响应）
    ↓
CORS 中间件（添加 CORS 头）
    ↓
响应返回
```

**注意：** 虽然 CORS 中间件先添加，但自定义中间件后添加，所以自定义中间件先执行。

## 🔄 两个中间件的协作

### 完整请求流程

```
1. 浏览器发送请求
   ↓
2. CORS 中间件处理预检请求（如果是 OPTIONS）
   ↓
3. 自定义中间件：
   - 生成 request_id
   - 记录开始时间
   - 记录开始日志
   ↓
4. 路由处理器执行业务逻辑
   ↓
5. 自定义中间件：
   - 计算处理时间
   - 添加响应头
   - 记录完成日志
   ↓
6. CORS 中间件添加 CORS 响应头
   ↓
7. 响应返回给浏览器
```

### 响应头完整示例

```http
HTTP/1.1 200 OK
Content-Type: application/json
Access-Control-Allow-Origin: *          ← CORS 中间件添加
Access-Control-Allow-Credentials: true   ← CORS 中间件添加
X-Process-Time: 2.3456                  ← 自定义中间件添加
X-Request-ID: 9fff2c98-509a-499f-95bc-9fd7a9b0442e  ← 自定义中间件添加

{
  "query": "...",
  "answer": "...",
  ...
}
```

## 📊 性能影响

### CORS 中间件
- **性能影响**：极小（仅添加响应头）
- **开销**：< 1ms

### 自定义中间件
- **性能影响**：极小
- **开销**：
  - UUID 生成：~0.001ms
  - 时间计算：~0.0001ms
  - 日志记录：~0.1-1ms（取决于日志系统）
  - **总开销**：< 2ms

**结论：** 两个中间件的性能开销都很小，不会影响 API 的整体性能。

## 🎯 总结

| 中间件 | 主要功能 | 适用场景 | 性能影响 |
|--------|----------|----------|----------|
| **CORS 中间件** | 解决跨域问题 | 前后端分离部署 | 极小（< 1ms） |
| **自定义 HTTP 中间件** | 请求追踪和性能监控 | 所有场景 | 极小（< 2ms） |

**两个中间件都是您代码的一部分，共同为 V006 系统提供：**
1. ✅ 跨域访问支持（CORS）
2. ✅ 请求追踪能力（Request ID）
3. ✅ 性能监控能力（Process Time）
4. ✅ 完整的日志记录

这些功能对于生产环境的 API 服务都是**必需的基础设施**。
